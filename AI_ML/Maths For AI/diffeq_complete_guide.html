Foundations & Core Concepts

The mathematical language that describes continuous change — and why it runs deep through modern AI.

A differential equation is an equation that relates a function to its own derivatives. Instead of asking "what is the value of y?", we ask "how does y change?" — and then integrate forward in time to find the answer. This perspective — of modeling dynamics rather than static states — is increasingly central to machine learning.

Why Differential Equations Appear in AI/ML

Machine learning is fundamentally about optimization over continuous landscapes, modeling how systems evolve, and understanding the flow of information. All three activities are described by differential equations:

  • Training dynamics: How do weights change as we apply gradient descent? The continuous limit is an ODE: dθ/dt = −∇L(θ).
  • Generative models: How do we move a noisy sample back to the data manifold? Diffusion models frame this as reversing a stochastic process (SDE).
  • Continuous architectures: What happens when ResNet layers become infinitely thin? You get a Neural ODE — a network whose depth is defined by integration time.
  • Physics-constrained learning: When you need predictions that obey conservation laws (energy, momentum, mass), you embed the governing PDE into the loss function.
📐

ODE — Ordinary

Derivatives with respect to one variable (usually time). Governs Neural ODEs, gradient flow, dynamical systems, and training trajectory analysis.

🌊

PDE — Partial

Multiple independent variables (space + time). Used in physics-informed neural networks, heat diffusion, fluid dynamics simulation, and image processing.

🎲

SDE — Stochastic

ODE plus a noise (Wiener process) term. The mathematical foundation of DDPM, score-based generative models, and Langevin MCMC sampling.

🧠

Neural ODE

A neural network parameterizes dh/dt. This enables continuous-depth models, irregular time series, and memory-efficient backpropagation via the adjoint method.

First-Order ODE — Standard Form

Initial Value Problem (IVP)
dy/dt = f(t, y), y(t₀) = y₀ Solution: y(t) = y₀ + ∫[t₀→t] f(s, y(s)) ds

Anatomy of the IVP

f(t, y) is the "right-hand side" — it defines the rate of change. In a Neural ODE, f is a neural network. In gradient flow, f is the negative gradient of the loss. In a diffusion model, f is a score function (gradient of log-probability).

y₀ is the initial condition — the starting point. In latent-space Neural ODEs, y₀ is encoded from input data. In generative models, y₀ is pure Gaussian noise.

The integral form shows that solving an ODE is fundamentally a continuous form of summation — exactly what happens when you run a network with many residual blocks stacked.

Real-World Applications

  • Continuous Normalizing Flows (CNF): The change-of-variables formula for probability densities uses the trace of the Jacobian of f, computable cheaply via Hutchinson's estimator. CNFs underpin generative models like FFJORD.
  • Protein folding dynamics: AlphaFold's confidence scores rely on understanding how protein structures evolve. Follow-up work models folding as ODE trajectories in conformation space.
  • Pharmacokinetics / drug dosing AI: Compartment models (ODE systems for drug concentration over time) are fitted to patient data using gradient-based optimization — standard ML tooling.
  • Recurrent neural network equivalence: A vanilla RNN in the limit of small time steps is exactly Euler integration of an ODE. Understanding this link allows better architectural choices.

# Real use: radioactive decay, learning rate schedules,

def exp_decay(t, y, k=0.5):

print(f"Max error: {np.max(np.abs(sol_decay.y[0] - y_exact)):.2e}")

# Real use: sigmoid activations (logistic ODE in discrete form),
#           user growth curves, epidemic modeling for analytics

    return [r * y[0] * (1 - y[0] / K)]

Systems of ODEs — The General Case

Why Neural Networks Use Systems of ODEs

A single ODE governs one variable. Real models have millions of coupled variables. A neural network with hidden state h ∈ ℝᵈ evolves as a system of d ODEs simultaneously — one per dimension. The magic is that the right-hand side f_θ: ℝᵈ → ℝᵈ is a learnable function (the network), shared across all dimensions and all time steps.

The SIR epidemiological model below is a good minimal example: three coupled ODEs for Susceptible, Infected, Recovered populations. Fitting β and γ to data is exactly what analytics dashboards do for disease modeling, customer churn, or viral product growth.

PYTHON sir_system.py — coupled ODEs + parameter fitting 
from scipy.optimize import minimize

def sir(t, y, beta, gamma):
    """
    S' = -beta*S*I/N    (susceptible becoming infected)
    I' =  beta*S*I/N - gamma*I  (infected recovery)
    R' =  gamma*I               (recovered)
    beta: transmission rate,  gamma: recovery rate
    """
    S, I, R = y
    N = S + I + R
    return [
        -beta * S * I / N,
         beta * S * I / N - gamma * I,
         gamma * I
    ]

def fit_sir_to_data(observed_I, t_obs, N=10_000):
    """Fit beta, gamma by minimising MSE against observed infections.
    Identical pattern to fitting any differentiable physics model to data."""
    def loss(params):
        beta, gamma = params
        if beta <= 0 or gamma <= 0: return 1e10
        sol = solve_ivp(
            lambda t, y: sir(t, y, beta, gamma),
            (t_obs[0], t_obs[-1]),
            [N-1, 1, 0], t_eval=t_obs, method='RK45'
        )
        return np.mean((sol.y[1] - observed_I)**2)

    res = minimize(loss, [0.3, 0.1], method='Nelder-Mead')
    return res.x  # [beta_fit, gamma_fit]

Numerical Methods from Scratch

Understanding integrators makes you better at debugging Neural ODEs, choosing tolerances, and seeing why ResNets work.

The Core Idea: Discretizing Continuous Change

We cannot compute the integral ∫f(t,y) dt analytically for most f. Numerical methods slice the time axis into small steps h and approximate the integral. The key trade-off is accuracy vs. cost: smaller steps → more accurate → more compute. Adaptive methods (RK45) automatically control step size so error stays below a user-specified tolerance (rtol/atol).

In deep learning, the step size h corresponds to the layer count (or layer width in ResNets). Euler = one residual block per step. RK4 = four function evaluations per block. That's why deep residual networks can be understood as ODE solvers — and why Neural ODEs are just the infinite-layer limit.

Euler's Method

Forward Euler — 1st order, global error O(h)
y_{n+1} = y_n + h · f(t_n, y_n) This IS a residual connection: y_{n+1} = y_n + F(y_n) → ResNet block with step size h = 1

Euler in ML Systems

  • ResNet = Euler ODE solver: He et al.'s residual connection y_{l+1} = y_l + F(y_l, W_l) is exactly one Euler step. This insight, formalized by E (2017) and Haber & Ruthotto, explains why ResNets can be very deep without vanishing gradients — stability of the ODE.
  • Euler-Maruyama for diffusion sampling: DDPM's reverse denoising process uses the stochastic Euler method (Euler-Maruyama). Each denoising step is one Euler integration of the reverse SDE.
  • Fast inference in diffusion models: DDIM (Denoising Diffusion Implicit Models) uses a more accurate (higher-order) ODE solver instead of Euler, reducing the number of function evaluations from 1000 to 50 while maintaining quality.

PYTHON euler_and_resnet.py — Euler integration + ResNet equivalence 
import numpy as np
import torch, torch.nn as nn

# ── Classical Euler integrator ───────────────────────────
def euler_solve(f, t0, y0, t_end, h):
    t, y = t0, np.array(y0, dtype=float)
    ts, ys = [t], [y.copy()]
    while t < t_end - 1e-10:
        h = min(h, t_end - t)
        y = y + h * np.asarray(f(t, y))
        t += h
        ts.append(t); ys.append(y.copy())
    return np.array(ts), np.array(ys)

# ── ResNet as Euler ODE (step size h=1) ──────────────────
# Each layer: h_{l+1} = h_l + F(h_l)  ← one Euler step
class ResNetAsODE(nn.Module):
    """Show ResNet is Euler integration of dh/dt = F(h)."""
    def __init__(self, dim, n_layers=10, h=1.0):
        super().__init__()
        self.h = h           # ODE step size
        self.layers = nn.ModuleList([
            nn.Sequential(nn.Linear(dim,dim), nn.ReLU(),
                           nn.Linear(dim,dim))
            for _ in range(n_layers)
        ])

    def forward(self, x):
        h = x
        for layer in self.layers:
            # Euler step: h_{t+1} = h_t + step_size * F(h_t)
            h = h + self.h * layer(h)
        return h

# As h→0 and n_layers→∞ (with n_layers * h = T fixed),
# this approaches a continuous Neural ODE!

Runge-Kutta 4 (RK4) — The Workhorse

RK4 — 4th order, global error O(h⁴)
k₁ = h·f(tₙ, yₙ) k₂ = h·f(tₙ + h/2, yₙ + k₁/2) k₃ = h·f(tₙ + h/2, yₙ + k₂/2) k₄ = h·f(tₙ + h, yₙ + k₃) y_{n+1} = yₙ + (k₁ + 2k₂ + 2k₃ + k₄) / 6

Why RK4 Matters for Neural ODEs

RK4 evaluates the derivative function f four times per step, using midpoint estimates to achieve 4th-order accuracy. In Neural ODE terms, this means calling the neural network 4 times per integration step — expensive but more accurate than Euler.

The Dormand-Prince RK45 method (used by solve_ivp default and dopri5 in torchdiffeq) embeds both 4th and 5th order estimates in 6 function evaluations, using the difference to control step size adaptively. This is what makes Neural ODEs powerful: they automatically use fewer steps in "easy" regions and more in "hard" ones.

def rk4_step(f, t, y, h):
    """Single RK4 step. Works with NumPy arrays or PyTorch tensors."""
    k1 = h * np.asarray(f(t,         y))
    k2 = h * np.asarray(f(t+h/2, y+k1/2))
    k3 = h * np.asarray(f(t+h/2, y+k2/2))
    k4 = h * np.asarray(f(t+h,   y+k3))
    return y + (k1 + 2*k2 + 2*k3 + k4) / 6

def rk4_solve(f, t0, y0, t_end, h):
    t, y = t0, np.asarray(y0, dtype=float)
    ts, ys = [t], [y.copy()]
    while t < t_end - 1e-10:
        h_eff = min(h, t_end-t)
        y = rk4_step(f, t, y, h_eff)
        t += h_eff
        ts.append(t); ys.append(y.copy())
    return np.array(ts), np.array(ys)

SciPy ODE Solvers in Practice

Production-grade ODE solving for scientific Python — parameter estimation, event detection, and stiff system handling.

Stiffness — The Hidden Enemy in ML Pipelines

A system is stiff when it contains components that evolve on vastly different timescales — some variables change extremely fast, others slowly. This forces explicit methods (RK45) to take tiny steps to remain stable, even in regions where the solution isn't changing much.

In ML contexts, stiffness arises in: neural ODE training with tight tolerances (the Jacobian of the network has both large and small eigenvalues), multi-scale physics simulations (fast acoustic waves + slow heat diffusion), and chemical reaction networks used in drug discovery. For stiff systems, always use method='Radau' or 'BDF' with an analytic Jacobian.

PYTHON scipy_full_options.py — stiff systems, Jacobians, events 
from scipy.integrate import solve_ivp
import numpy as np

# ── Van der Pol oscillator (stiff for large mu) ──────────
def van_der_pol(t, y, mu=1000.0):
    return [y[1], mu*(1-y[0]**2)*y[1]-y[0]]

def vdp_jac(t, y, mu=1000.0):
    """Analytical Jacobian — ALWAYS provide for stiff solvers.
    Speeds up implicit methods by 10-100x vs finite differences."""
    return [[0, 1],
             [-2*mu*y[0]*y[1]-1, mu*(1-y[0]**2)]]

sol = solve_ivp(
    van_der_pol, (0, 3000), [2.0, 0.0],
    method='Radau',       # implicit, stiff-safe
    jac=vdp_jac,          # analytical Jacobian
    rtol=1e-4, atol=1e-6,
    dense_output=True    # for continuous-time interpolation
)

# ── Event detection: stop when hidden state crosses threshold
# Real use: early stopping in Neural ODE forward pass,
#           detecting stability transitions in training dynamics
def threshold_event(t, y): return y[0] - 0.1  # trigger when y[0] = 0.1
threshold_event.terminal  = True
threshold_event.direction = -1

sol_event = solve_ivp(

    method='Radau'

print(f"Stopped at t = {sol_event.t[-1]:.3f}")

Neural ODEs

The continuous-depth network — where depth becomes a continuous variable and backprop becomes solving an ODE backwards.

What Makes a Neural ODE Different

In a standard residual network: h_{l+1} = h_l + F_l(h_l), where each layer l has its own parameters. A Neural ODE replaces this with: dh/dt = f_θ(h(t), t), where θ is a single shared parameter set and t is a continuous depth variable.

Key theoretical properties:

  • Memory efficiency: Instead of storing all intermediate activations for backprop (O(L) memory), the adjoint method recomputes them by solving the ODE backwards, using O(1) memory regardless of depth.
  • Adaptive computation: The ODE solver uses more steps in "hard" regions (near sharp transitions) and fewer in "easy" regions — like dynamic compute allocation per sample.
  • Continuous normalizing flows: By tracking the log-determinant of the Jacobian alongside h, Neural ODEs enable exact likelihood computation in generative models.
  • Irregular time series: Since the ODE operates in continuous time, it naturally handles observations at arbitrary, non-uniform timestamps — impossible for standard LSTMs/Transformers without interpolation.
Neural ODE — Forward + Adjoint Backward
Forward: h(t₁) = h(t₀) + ∫[t₀→t₁] f_θ(h(t), t) dt (ODE solve) Adjoint: a(t) = dL/dh(t) (adjoint state) da/dt = -a(t)ᵀ · ∂f/∂h (adjoint ODE, solved backwards) dL/dθ = -∫[t₁→t₀] a(t)ᵀ · ∂f/∂θ dt (gradient w.r.t. params) Memory: O(1) vs O(L) for standard backprop-through-layers
PYTHON · PYTORCH neural_ode_full.py — classifier + latent ODE for time series 
import torch, torch.nn as nn
from torchdiffeq import odeint_adjoint as odeint

# ── Dynamics network ──────────────────────────────────────
class ODEFunc(nn.Module):
    """
    Parameterizes dh/dt = f_θ(h, t).
    Design choices:
      - Use Tanh (bounded activations → stable dynamics)
      - Small init weights (start near identity transform)
      - No BatchNorm (breaks ODE continuity in t)
    """
    def __init__(self, dim=64):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(dim+1, 128),   # +1 for time t concatenated
            nn.Tanh(),
            nn.Linear(128, 128),
            nn.Tanh(),
            nn.Linear(128, dim),
        )
        for m in self.net.modules():
            if isinstance(m, nn.Linear):
                nn.init.normal_(m.weight, 0, 0.01)

    def forward(self, t, h):
        t_expand = t.expand(h.shape[0], 1)
        return self.net(torch.cat([h, t_expand], dim=1))

# ── Full classifier ───────────────────────────────────────
class NeuralODEClassifier(nn.Module):
    def __init__(self, in_dim, hidden=64, n_classes=10):
        super().__init__()
        self.encoder    = nn.Linear(in_dim, hidden)
        self.ode_func   = ODEFunc(hidden)
        self.norm       = nn.LayerNorm(hidden)
        self.classifier = nn.Linear(hidden, n_classes)
        self.t_span     = torch.tensor([0.0, 1.0])

    def forward(self, x):
        h0 = self.encoder(x)
        # odeint_adjoint: O(1) memory backprop via adjoint method
        h1 = odeint(
            self.ode_func, h0,
            self.t_span.to(x.device),
            method='dopri5',       # adaptive RK45
            rtol=1e-3, atol=1e-4,
            adjoint_params=list(self.ode_func.parameters())
        )[-1]
        return self.classifier(self.norm(h1))

# ── Latent ODE for irregular time series ─────────────────
class LatentODE(nn.Module):
    """
    Encode sparse/irregular observations → initial latent z₀
    → evolve z continuously via ODE → decode at any query times.
    Used for: ICU prediction, financial time series, IoT sensors.
    """
    def __init__(self, obs_dim, latent_dim=32):
        super().__init__()
        self.rnn_encoder = nn.GRU(obs_dim+1, latent_dim, batch_first=True)
        self.z0_net      = nn.Linear(latent_dim, latent_dim*2)
        self.ode_func    = ODEFunc(latent_dim)
        self.decoder     = nn.Linear(latent_dim, obs_dim)

    def forward(self, x_obs, t_obs, t_pred):
        # Concatenate times with observations for encoder
        x_with_t = torch.cat([x_obs, t_obs.unsqueeze(-1)], -1)
        _, h = self.rnn_encoder(x_with_t.flip(1))  # encode backwards
        z0_params = self.z0_net(h.squeeze(0))
        mu, logstd = z0_params.chunk(2, dim=-1)
        z0 = mu + torch.randn_like(mu) * logstd.exp()  # reparameterize
        z_traj = odeint(self.ode_func, z0, t_pred)    # [T, B, D]
        return self.decoder(z_traj), mu, logstd

Physics-Informed Neural Networks (PINNs)

Embedding physical laws directly into the loss function — no mesh, no discretization, just neural networks that respect the equations.

The Core Idea: Residual as Loss

A traditional numerical solver (FEM, FDM) discretizes space into a mesh and enforces the PDE at each grid point. PINNs take a radically different approach: train a neural network u_θ(x,t) to approximate the solution, and penalize violations of the PDE directly in the loss function.

The PDE residual is computed using automatic differentiation — the same mechanism as backprop. For ∂u/∂x, we call torch.autograd.grad(u.sum(), x). This means no finite differences, no mesh, and the solution is a smooth function valid everywhere in the domain.

Three loss terms are combined: L_pde (residual at random interior collocation points), L_bc (boundary conditions), and L_data (any observed measurements). The balance between them (λ weights) is crucial — poor weighting is the #1 failure mode in PINNs.

PINN Loss — General Form
L_total = λ_pde · (1/N_c) Σᵢ |N[u_θ](xᵢ,tᵢ)|² + λ_bc · (1/N_b) Σⱼ |u_θ(xⱼ,tⱼ) - ubc(xⱼ,tⱼ)|² + λ_ic · (1/N_i) Σₖ |u_θ(xₖ,t₀) - u₀(xₖ)|² + λ_data· (1/N_d) Σₗ |u_θ(xₗ,tₗ) - ûₗ|² where N[u] is the PDE operator (e.g., ∂u/∂t + u·∂u/∂x - ν·∂²u/∂x² for Burgers)

import torch, torch.nn as nn

class PINN(nn.Module):
    """
    PINN for Burgers' equation: ∂u/∂t + u·∂u/∂x = ν·∂²u/∂x²
    Input: (x, t) ∈ [-1,1] × [0,1]   Output: u(x,t)
    Architecture notes:
      - Tanh activation (smooth → higher-order derivatives exist)
      - Xavier init (balanced gradient flow through deep network)
      - No dropout (stochasticity breaks PDE constraint training)
    """
    def __init__(self, layers=[2,64,64,64,64,1]):
        super().__init__()
        net = []
        for i in range(len(layers)-1):
            net += [nn.Linear(layers[i], layers[i+1])]
            if i < len(layers)-2: net += [nn.Tanh()]
        self.net = nn.Sequential(*net)
        for m in self.net.modules():
            if isinstance(m, nn.Linear):
                nn.init.xavier_normal_(m.weight)

    def forward(self, x, t):
        return self.net(torch.cat([x, t], -1))

def burgers_residual(model, x, t, nu=0.01/3.14159):
    """Compute PDE residual via autograd — no finite differences!"""
    u   = model(x, t)
    # ∂u/∂x and ∂u/∂t — create_graph=True enables higher-order grads
    u_x = torch.autograd.grad(u.sum(), x, create_graph=True)[0]
    u_t = torch.autograd.grad(u.sum(), t, create_graph=True)[0]
    # ∂²u/∂x² — second derivative
    u_xx= torch.autograd.grad(u_x.sum(), x, create_graph=True)[0]
    res = u_t + u*u_x - nu*u_xx    # Burgers PDE residual
    return (res**2).mean()

SDEs & Diffusion Models

The mathematical bridge between Brownian motion and state-of-the-art generative AI — DALL-E, Stable Diffusion, Sora, and beyond.

From SDE to Generative Model

A Stochastic Differential Equation (SDE) is an ODE with an additional noise term driven by a Wiener process (Brownian motion W_t):

dx = f(x,t) dt + g(t) dW

The insight of Song et al. (NeurIPS 2020, "Score-Based Generative Modeling through SDEs") is profound: any SDE that transforms data into noise has a time-reversed SDE that transforms noise back into data. This reverse SDE requires knowing the score function ∇_x log p_t(x) — the gradient of the log-probability density at time t. A neural network trained to predict this score function is a score model, and running the reverse SDE gives you a generative model.

DDPM (Ho et al.) is a discrete special case. The continuous SDE view unifies all modern diffusion models and enables using better ODE/SDE solvers for faster sampling.

Forward + Reverse SDE (Song et al. 2020)
Forward (add noise): dx = f(x,t) dt + g(t) dW Reverse (denoise): dx = [f(x,t) - g(t)² · ∇_x log p_t(x)] dt + g(t) dW̄ VP-SDE (DDPM): f = -½β(t)x, g = √β(t) VE-SDE (NCSN): f = 0, g = √(dσ²(t)/dt) Prob-flow ODE: dx = [f(x,t) - ½g(t)² · ∇_x log p_t(x)] dt (deterministic)

import torch, torch.nn as nn
from scipy.integrate import solve_ivp

class VPSDE:
    """Variance Preserving SDE — continuous DDPM formulation."""
    def __init__(self, beta_min=0.1, beta_max=20.0, T=1.0):
        self.beta_min = beta_min; self.beta_max = beta_max; self.T = T

    def beta(self, t):
        return self.beta_min + t*(self.beta_max-self.beta_min)

    def marginal_prob(self, x0, t):
        """x_t ~ N(mean, std²·I)  — forward diffusion closed form"""
        log_coeff = -0.25*t**2*(self.beta_max-self.beta_min) - 0.5*t*self.beta_min
        mean = x0 * log_coeff.exp()
        std  = ((1 - (2*log_coeff).exp()).clamp(0)).sqrt()
        return mean, std

    def score_matching_loss(self, score_net, x0):
        """Denoising score matching — predict ε given (x_t, t)."""
        t    = torch.rand(x0.size(0))*0.999+0.001
        t4d  = t.view(-1,1,1,1)
        mean, std = self.marginal_prob(x0, t4d)
        eps  = torch.randn_like(x0)
        x_t  = mean + std * eps
        return ((score_net(x_t, t) - eps)**2).mean()

    def euler_maruyama_sample(self, score_net, shape, n_steps=1000):
        """Reverse-time SDE sampling via Euler-Maruyama."""
        x = torch.randn(shape)  # start from noise
        ts = torch.linspace(self.T, 1e-3, n_steps+1)
        for i in range(n_steps):
            t   = ts[i] * torch.ones(shape[0])
            dt  = ts[i+1] - ts[i]            # negative (going backwards)
            g   = self.beta(t).sqrt().view(-1,1,1,1)
            s   = score_net(x, t)             # predicted score
            f   = -0.5 * self.beta(t).view(-1,1,1,1) * x
            # Reverse SDE: dx = (f - g²·score)·dt + g·dW
            x   = x + (f - g**2 * s) * dt + g * (-dt).sqrt() * torch.randn_like(x)
        return x.clamp(-1, 1)

    def ode_sample(self, score_net, shape, rtol=1e-4):
        """Probability flow ODE sampler — deterministic, faster.
        Same marginals as SDE but allows using high-order ODE solvers."""
        x0 = torch.randn(shape).flatten().numpy()
        def ode_fn(t, x_flat):
            x = torch.tensor(x_flat).reshape(shape).float()
            t_t = torch.full((shape[0],), t)
            with torch.no_grad():
                s = score_net(x, t_t)
            beta_t = self.beta(torch.tensor(t))
            # Probability flow ODE (no noise term)
            dx = (-0.5*beta_t*x - 0.5*beta_t*s).flatten().numpy()
            return dx
        sol = solve_ivp(ode_fn, (self.T, 1e-3), x0,
                         method='RK45', rtol=rtol)
        return torch.tensor(sol.y[:-1]).reshape(shape)

Gradient Flow & Optimization Dynamics

Every training run is secretly solving an ODE — understanding gradient flow explains generalization, convergence, and why some architectures train better.

Training as an ODE

Gradient descent is a numerical ODE integrator. When you update θ ← θ - η·∇L(θ), you're applying one Euler step with step size η to the gradient flow ODE:

dθ/dt = -∇_θ L(θ)

This ODE perspective has deep consequences. The learning rate η is the step size of the ODE integrator — too large and the Euler method diverges (unstable training). Momentum (SGD with momentum, Adam) corresponds to a second-order ODE (Polyak's heavy ball): d²θ/dt² + γ·dθ/dt = -∇L(θ). This is why momentum "overshoots" minima and why Nesterov's method (a more accurate ODE integrator for this second-order system) works better.

Weight decay adds a linear restoring force: dθ/dt = -∇L - λθ. The steady state (where dθ/dt = 0) satisfies ∇L + λθ = 0 — a regularized minimum.

Gradient Flow — Optimizer Hierarchy
SGD: dθ/dt = -∇L(θ) [1st order ODE] Momentum: d²θ/dt² = -∇L(θ) - γ·dθ/dt [2nd order ODE, damped oscillator] Adam: dθ/dt ≈ -m/(√v + ε) where m,v are EWMAs [preconditioned flow] Weight decay: dθ/dt = -∇L(θ) - λθ [restoring force = L2 reg] Langevin: dθ = -∇L(θ)dt + √(2T)·dW [SDE — stochastic gradient descent]

# The stability of gradient flow depends on eigenvalues of ∇²L (Hessian)
# Euler method is stable iff: |1 - lr * λ_max| ≤ 1
# → max stable lr = 2 / λ_max  (Courant condition for gradient descent!)

def max_stable_lr(model, loss_fn, x, y, n_power_iter=20):
    """Estimate max stable learning rate via power iteration on Hessian."""
    params = list(model.parameters())
    loss   = loss_fn(model(x), y)
    grads  = torch.autograd.grad(loss, params, create_graph=True)
    v      = [torch.randn_like(p) for p in params]
    for _ in range(n_power_iter):
        Hv = torch.autograd.grad(grads, params, grad_outputs=v, retain_graph=True)
        lambda_max = sum((hv*vi).sum() for hv,vi in zip(Hv,v))
        v_norm     = (sum(vi.norm()**2 for vi in v).sqrt())
        v          = [hv/v_norm for hv in Hv]
    return 2.0 / lambda_max.item()   # max stable learning rate

Gradient Flow Theory — LLM Training Impact

  • Learning rate schedule design for LLMs: The "warmup then cosine decay" schedule used to train GPT, LLaMA, and Gemini is motivated by gradient flow stability — at init, the Hessian eigenvalue λ_max is large, so lr must be small. Warmup gradually increases lr as the network settles into a stable region.
  • Edge of Stability (Cohen et al., 2021): Training large LLMs at high lr causes loss oscillations but still converges — explained by gradient flow analysis showing that adaptive methods like Adam self-regulate around the stability boundary 2/λ_max.
  • Muon optimizer (gradient flow perspective): Muon (Momentum + orthogonalization) can be derived from the gradient flow ODE with a Riemannian metric on the weight manifold — better geometry → faster convergence. Used at Modular AI and reported to outperform AdamW on some LLM benchmarks.
  • LoRA / PEFT via gradient flow: Low-rank adaptation constrains the gradient flow to a low-dimensional subspace of parameter space, dramatically reducing the compute needed for fine-tuning. Understanding which subspace captures the important curvature requires Hessian analysis of the full gradient flow.
  • Neural Tangent Kernel (NTK): In the infinite-width limit, gradient flow linearizes: the network parameters barely move, and training is described exactly by an ODE with the NTK as the kernel. NTK theory predicts generalization and is used to analyze transformer attention patterns.

Differential Equations in LLM Internals

Transformers, attention, and token generation through the lens of continuous dynamical systems.

Transformers as Interacting Particle Systems

A transformer layer can be viewed as one step of a differential equation over token representations. Geshkovski et al. (2023) showed that as depth → ∞, the transformer forward pass solves an interacting particle system ODE:

dxᵢ/dt = Σⱼ softmax(xᵢᵀxⱼ/√d) · (xⱼ - xᵢ)

This is a mean-field aggregation ODE — each token flows toward an attention-weighted average of all other tokens. In the long-time limit (very deep networks), all tokens collapse to a cluster — the "rank collapse" problem. This ODE perspective explains why depth normalization (LayerNorm, RMSNorm) and residual connections are essential for stability.

The feedforward network layer then acts as a second ODE that repels tokens from each other in a feature-selective way, preventing full collapse. Together, attention + FFN create a competitive dynamical system that preserves information while mixing representations.

Transformer Layer as ODE Step (Geshkovski et al. 2023)
dxᵢ/dt = Σⱼ A(xᵢ, xⱼ) · (xⱼ - xᵢ) [attention = aggregation ODE] where A(xᵢ, xⱼ) = softmax(xᵢᵀWQᵀWKxⱼ/√d) [attention weights] Full layer (with residual): x ← x + Attn(LN(x)) [one Euler step of aggregation ODE] x ← x + FFN(LN(x)) [one Euler step of repulsion ODE]

import torch, torch.nn as nn, torch.nn.functional as F
from torchdiffeq import odeint_adjoint as odeint

# ── Attention aggregation as ODE right-hand side ──────────
class AttentionODEFunc(nn.Module):
    """
    Implements dx/dt = Σⱼ softmax(xᵢᵀxⱼ/√d) * (xⱼ - xᵢ)
    Continuous limit of transformer self-attention layers.
    Theoretical insight: tokens flow toward their attention-weighted mean.
    """
    def __init__(self, d_model, n_heads=8):
        super().__init__()
        self.d_model  = d_model
        self.n_heads  = n_heads
        self.d_head   = d_model // n_heads
        self.Wq = nn.Linear(d_model, d_model, bias=False)
        self.Wk = nn.Linear(d_model, d_model, bias=False)
        self.Wv = nn.Linear(d_model, d_model, bias=False)
        self.ffn = nn.Sequential(
            nn.Linear(d_model, 4*d_model), nn.GELU(),
            nn.Linear(4*d_model, d_model)
        )
        self.norm1 = nn.LayerNorm(d_model)
        self.norm2 = nn.LayerNorm(d_model)

    def forward(self, t, x):
        # x: [B, L, D]   t: scalar (integration time = depth)
        B, L, D = x.shape
        h   = self.norm1(x)
        q   = self.Wq(h).view(B,L,self.n_heads,self.d_head).transpose(1,2)
        k   = self.Wk(h).view(B,L,self.n_heads,self.d_head).transpose(1,2)
        v   = self.Wv(h).view(B,L,self.n_heads,self.d_head).transpose(1,2)
        attn= F.scaled_dot_product_attention(q,k,v)   # Flash-Attention compatible
        attn= attn.transpose(1,2).reshape(B,L,D)
        # Aggregation ODE: attn(x) - x represents dx/dt
        dx_attn = attn - x                       # flow toward avg
        dx_ffn  = self.ffn(self.norm2(x + dx_attn)) # repulsion
        return dx_attn + dx_ffn

# ── Continuous-depth transformer ──────────────────────────
class ContinuousDepthTransformer(nn.Module):
    """
    Replace N discrete transformer layers with a Neural ODE.
    Advantages:
      - O(1) memory for backprop via adjoint method
      - Adaptive compute: harder sequences → more integration steps
      - Depth is a real-valued hyperparameter, not integer
    """
    def __init__(self, vocab_size, d_model=256, max_depth=1.0):
        super().__init__()
        self.embed    = nn.Embedding(vocab_size, d_model)
        self.ode_func = AttentionODEFunc(d_model)
        self.out_proj = nn.Linear(d_model, vocab_size)
        self.t_depth  = torch.tensor([0.0, max_depth])

    def forward(self, input_ids):
        x = self.embed(input_ids)      # [B, L, D]
        # Integrate: x evolves continuously from depth 0 → max_depth
        x = odeint(self.ode_func, x,
                     self.t_depth.to(x.device),
                     method='dopri5', rtol=1e-3)[-1]
        return self.out_proj(x)       # [B, L, vocab_size]

ODE/PDE Theory Applied to LLMs

  • Mixture of Depths (MoD): Adaptive depth allocation in LLMs — some tokens exit the ODE early (low depth), others continue. This is analogous to adaptive step size control in ODE solvers, and is used in models like Mistral and in speculative decoding research.
  • Attention sink phenomenon: The first token ("sink token") accumulates anomalously high attention weights. ODE analysis shows this is a stable fixed point of the aggregation dynamics — all particle trajectories eventually pass through. StreamingLLM exploits this to enable infinite-context generation.
  • Chain-of-thought as integration time: Interpretability research (Anthropic, DeepMind) models reasoning as the LLM needing more "integration time" (more tokens / depth) to resolve ambiguity — analogous to a stiff ODE needing smaller step sizes near sharp transitions.
  • KV-cache as ODE state: The KV cache in transformer inference stores the evolved state of key/value projections — essentially caching the ODE trajectory so autoregressive generation doesn't re-integrate from scratch at each step.
  • Knowledge editing via ODE perturbation: ROME (Rank-One Model Editing) and MEMIT modify specific factual associations in LLMs. ODE analysis frames this as perturbing a stable fixed point of the forward-pass dynamics — too large a perturbation and the model loses coherence (instability).

Analytics Applications

From forecasting and survival analysis to causal modeling — differential equations power the quantitative backbone of modern data analytics.

Why Differential Equations Belong in Analytics

Data analytics traditionally uses static regression or black-box ML. Differential equations bring three advantages: interpretability (the rate equation directly describes the business mechanism), sample efficiency (physics constraints reduce the effective degrees of freedom), and extrapolation (ODE models can forecast beyond the training time range if the mechanism is correctly specified).

The pattern is always the same: identify the mechanism as a differential equation, embed unknown terms as neural networks (Universal Differential Equations), and train end-to-end on data. The neural terms learn residuals that the equation doesn't capture.

Survival Analysis — The Hazard ODE

Survival Analysis as ODE
dS/dt = -h(t) · S(t) where S(t) = P(survive past t), h(t) = hazard rate Solution: S(t) = exp(-∫₀ᵗ h(s) ds) = exp(-H(t)) [cumulative hazard] h(t) can be a neural network: DeepHit, Cox-Time, etc.

import torch, torch.nn as nn
import numpy as np
from scipy.integrate import solve_ivp, quad

# ── 1. Neural hazard function (survival analysis) ─────────
# Real use: customer lifetime value (CLV), churn prediction,
#           clinical trial analysis, equipment failure AI
class NeuralHazard(nn.Module):
    """h_θ(t, x) — neural parameterization of the hazard rate.
    x: static features (demographics, product usage, etc.)"""
    def __init__(self, n_features, hidden=64):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(n_features+1, hidden), nn.ReLU(),
            nn.Linear(hidden, hidden),    nn.ReLU(),
            nn.Linear(hidden, 1),          nn.Softplus()  # h > 0
        )

    def hazard(self, t, x):
        t_feat = t * torch.ones(x.size(0), 1)
        return self.net(torch.cat([x, t_feat], -1)).squeeze(-1)

    def survival(self, t_eval, x):
        """S(t|x) = exp(-∫₀ᵗ h(s,x) ds)  — solve ODE for each x"""
        def dS_dt(t, S_flat):  # S_flat: [batch]
            h = self.hazard(t, x).detach().numpy()
            return -h * S_flat
        sol = solve_ivp(dS_dt, (0, t_eval.max()),
                         np.ones(x.size(0)),  # S(0)=1
                         t_eval=t_eval.numpy())
        return torch.tensor(sol.y.T)  # [batch, n_times]

# ── 2. Universal Differential Equations (UDE) ────────────
# Known mechanism + neural network for unknown terms
# Real use: demand forecasting, financial modeling, process control
class UDE(nn.Module):
    """
    dx/dt = known_physics(x, t) + neural_residual(x, t)
    Example: sales growth with known seasonal trend + learned noise
    """
    def __init__(self):
        super().__init__()
        self.neural_term = nn.Sequential(
            nn.Linear(2, 32), nn.Tanh(), nn.Linear(32, 1)
        )

    def forward(self, t, x):
        # Known part: seasonal logistic growth
        K = 1000.0
        r_seasonal = 0.1 * (1 + 0.3 * torch.sin(2*3.14159*t/365))
        known = r_seasonal * x * (1 - x / K)
        # Neural residual: learns promotions, market shocks, etc.
        feat   = torch.stack([t.expand_as(x), x], -1)
        neural = self.neural_term(feat).squeeze(-1)
        return known + 0.1 * neural  # balance scales

# SEIR variant: Susceptible → Exposed → Engaged → Retained
# Fitted to user funnel data to optimize marketing spend
def user_funnel_ode(t, y, beta, sigma, gamma, delta):
    S, E, I, R = y     # Susceptible, Exposed, Engaged, Retained
    N = S+E+I+R
    return [
        -beta*S*I/N,                  # S' = outflow to Exposed
         beta*S*I/N - sigma*E,        # E' = Exposed → Engaged
         sigma*E     - gamma*I,        # I' = Engaged  → Retained/Churn
         gamma*I     - delta*R         # R' = Retained → Churn
    ]

# Unbiased trace estimator: Tr(J) ≈ εᵀ(Jε) for random ε ~ N(0,I)
def cnf_step(f_net, t, state):
    z, log_p = state
    eps = torch.randn_like(z)
    with torch.enable_grad():
        z_ = z.requires_grad_(True)
        fz = f_net(z_, t)
        # Hutchinson trace estimator (cheaper than full Jacobian)
        e_Jv = torch.autograd.grad(fz, z_, grad_outputs=eps,
                                         create_graph=True)[0]
        div_f = (e_Jv * eps).sum(-1)
    return fz, -div_f   # [dz/dt, d(log p)/dt]

JAX & Diffrax — High-Performance Differentiable ODEs

JIT-compiled, vectorized, and fully differentiable ODE solving — the backbone of modern research-scale neural differential equations.

Why JAX Changes the Game for Neural ODEs

PyTorch's torchdiffeq requires the adjoint method for memory efficiency because PyTorch autograd stores computation graphs eagerly. JAX uses functional transformations instead:

  • jax.jit: Traces and compiles the entire ODE solve (including the solver loop) into XLA. A Neural ODE forward pass becomes a single compiled kernel — 5–20× faster than PyTorch on hardware accelerators.
  • jax.vmap: Vectorizes the ODE solve over a batch dimension without writing a batch loop. Each example in the batch can have different integration times — impossible with standard PyTorch batching.
  • jax.grad through ODE: JAX differentiates through the solver's internal control flow, including adaptive step-size selection. This is "through-the-solver" differentiation without the adjoint trick.
  • Checkpointing: diffrax.BacksolveAdjoint gives the same O(1) memory as torchdiffeq's adjoint, but composable with vmap and jit.

The diffrax library (Kidger, 2021) provides all major solvers (Euler, Heun, Dopri5, Tsit5, ImplicitEuler, Kvaerno5) in a unified, differentiable JAX API. It's the reference implementation for continuous normalizing flows and flow matching in JAX-based research.

def solve_single(model, y0, t0=0.0, t1=1.0):
    term   = ODETerm(model)
    solver = Tsit5()                     # Tsitouras 5th-order (faster than dopri5)
    ctrl   = PIDController(rtol=1e-3, atol=1e-4)  # adaptive step

        term, solver, t0, t1, dt0=0.1,
        y0=y0, stepsize_controller=ctrl,
        saveat=save,
        adjoint=BacksolveAdjoint()     # O(1) memory adjoint
    )
    return sol.ys[-1]                  # shape: [dim]

    return model, opt_state, loss

Flow Matching & Rectified Flows

The next generation of generative models — simpler training than diffusion, straighter trajectories, fewer sampling steps.

From Diffusion to Flow Matching

Diffusion models add noise via a complex SDE schedule and train a score network to reverse it. Flow Matching (Lipman et al., 2022; Liu et al., 2022) takes a simpler approach: define a probability path p_t(x) that interpolates between the data distribution p_data and a simple prior p_noise, then train a neural network to predict the velocity field v_θ(x, t) that transports samples along this path.

The key advantage: Flow Matching constructs straight-line trajectories (Rectified Flow) from noise to data, while diffusion trajectories curve. Straight trajectories need far fewer ODE steps to integrate accurately — enabling high-quality generation in 1–4 steps instead of 20–1000.

The training objective is simply an MSE between the network's predicted velocity and the target velocity at each interpolated point — no score function, no SDE reversal, no complex weighting schedules.

Flow Matching — Conditional Flow Matching (CFM) Objective
Path: x_t = (1-t)·x_noise + t·x_data (linear interpolation, t ∈ [0,1]) Target: u_t(x_t | x_data) = x_data - x_noise (constant target velocity!) Loss: L_CFM(θ) = E_{t,x_data,x_noise} [ ||v_θ(x_t, t) - u_t||² ] Inference: dx/dt = v_θ(x, t), x(0) ~ N(0,I) → x(1) ~ p_data (Solve ODE forward from noise to data using any ODE solver)

import torch, torch.nn as nn
from scipy.integrate import solve_ivp
import numpy as np

# ── Velocity network (same architecture as diffusion U-Net) ──
class VelocityNet(nn.Module):
    """v_θ(x, t) → predicted velocity at time t.
    For images: use U-Net. For 2D toy data: MLP."""
    def __init__(self, dim=2, hidden=256):
        super().__init__()
        # Time embedding (sinusoidal, same as diffusion transformers)
        self.time_embed = nn.Sequential(
            nn.Linear(1, hidden), nn.SiLU(), nn.Linear(hidden, hidden)
        )
        self.net = nn.Sequential(
            nn.Linear(dim + hidden, hidden), nn.SiLU(),
            nn.Linear(hidden, hidden),       nn.SiLU(),
            nn.Linear(hidden, hidden),       nn.SiLU(),
            nn.Linear(hidden, dim)
        )

    def forward(self, x, t):
        # t: [B] scalar time per sample
        t_emb = self.time_embed(t.unsqueeze(-1))       # [B, hidden]
        return self.net(torch.cat([x, t_emb], -1))

# ── Conditional Flow Matching training step ───────────────
def cfm_loss(velocity_net, x_data):
    """
    Optimal Transport CFM (OT-CFM):
      x_t = (1-t) * x_noise + t * x_data     (straight-line path)
      target_v = x_data - x_noise             (constant velocity)
    """
    B, D = x_data.shape
    t       = torch.rand(B, device=x_data.device)        # sample t ~ Uniform[0,1]
    x_noise = torch.randn_like(x_data)                  # sample noise

    # Linear interpolation (straight path from noise → data)
    x_t     = (1 - t.unsqueeze(1)) * x_noise + t.unsqueeze(1) * x_data

    # Target velocity: constant along the straight path
    target_v = x_data - x_noise

    # Network predicts velocity; MSE loss
    pred_v  = velocity_net(x_t, t)
    return ((pred_v - target_v)**2).mean()    # simple MSE — no reweighting!

# ── Sampling via ODE integration (noise → data) ───────────
def cfm_sample(velocity_net, n_samples, dim, n_steps=50):
    """
    Solve dx/dt = v_θ(x, t) from t=0 (noise) to t=1 (data).
    Use any ODE solver — Euler, Heun, RK4, dopri5.
    Fewer steps needed because paths are nearly straight.
    """
    x0 = torch.randn(n_samples, dim)  # start from Gaussian noise
    ts = torch.linspace(0, 1, n_steps+1)
    x  = x0
    with torch.no_grad():
        for i in range(n_steps):
            t  = ts[i] * torch.ones(n_samples)
            dt = ts[i+1] - ts[i]
            v  = velocity_net(x, t)         # predicted velocity
            x  = x + dt * v                 # Euler step (50 steps is usually enough)
    return x  # shape: [n_samples, dim]

# ── Classifier-free guidance for Flow Matching ────────────
def cfg_velocity(vel_net, x, t, cond, cfg_scale=3.0):
    """Classifier-free guidance: amplify conditioned direction.
    Used in Stable Diffusion 3, FLUX, and other production FM models."""
    v_uncond = vel_net(x, t, cond=None)
    v_cond   = vel_net(x, t, cond=cond)
    return v_uncond + cfg_scale * (v_cond - v_uncond)

Stochastic Gradient Langevin Dynamics (SGLD)

Bayesian deep learning meets SDE — injecting noise into gradient descent to sample from the posterior over parameters.

Langevin Dynamics — Gradient Descent with Noise

Standard SGD minimizes the loss and converges to a point estimate θ*. Stochastic Gradient Langevin Dynamics (Welling & Teh, 2011) adds calibrated Gaussian noise to each update, turning gradient descent into a Markov Chain Monte Carlo (MCMC) sampler:

dθ = -∇L(θ) dt + √(2T) dW

The key insight: this SDE has the posterior p(θ|data) ∝ exp(-L(θ)/T) as its stationary distribution. Running SGLD long enough samples from the Bayesian posterior over parameters — giving uncertainty estimates for free, without variational inference or separate networks.

The temperature T controls the exploration-exploitation trade-off. T=0 is standard SGD. T=1 gives the exact posterior. T>1 over-spreads the distribution. Cyclical SGLD (Zhang et al., 2020) cycles T between high (exploration) and low (sampling) to collect diverse posterior samples efficiently.

SGLD Update Rule & Posterior Sampling
SGLD step: θ_{t+1} = θ_t - ε_t · ∇L̃(θ_t) + √(2ε_t · T) · η_t where η_t ~ N(0, I), ε_t → 0 (decaying step size) Stationary distribution: π(θ) ∝ exp(-L(θ)/T) = p(θ|data)^(1/T) (p(θ|data) for T=1, flat distribution for T→∞) Uncertainty estimate: Var[f(x)] ≈ (1/K) Σₖ f(x; θₖ)² - ((1/K)Σₖ f(x;θₖ))² where {θ₁,...,θK} are collected SGLD samples

import torch, torch.nn as nn
import numpy as np
from collections import deque

class SGLDOptimizer(torch.optim.Optimizer):
    """
    Stochastic Gradient Langevin Dynamics optimizer.
    Drop-in replacement for SGD that samples from the posterior.
    Key difference from SGD: adds noise scaled by √(2·lr·T).
    """
    def __init__(self, params, lr=1e-3, temperature=1.0,
                 weight_decay=0.0):
        defaults = dict(lr=lr, temperature=temperature,
                         weight_decay=weight_decay)
        super().__init__(params, defaults)

    @torch.no_grad()
    def step(self):
        for group in self.param_groups:
            lr   = group['lr']
            T    = group['temperature']
            wd   = group['weight_decay']
            noise_std = ((2 * lr * T)**0.5)  # Langevin noise scale
            for p in group['params']:
                if p.grad is None: continue
                grad = p.grad
                if wd != 0:
                    grad = grad + wd * p      # L2 regularization = Gaussian prior
                # Standard gradient step + Langevin noise
                p.add_(-lr * grad + noise_std * torch.randn_like(p))

class BayesianEnsemble:
    """
    Collect parameter snapshots during SGLD run.
    Ensemble gives calibrated uncertainty estimates.
    Real use: uncertainty-aware predictions in medical AI,
              active learning, out-of-distribution detection.
    """
    def __init__(self, model, max_snapshots=20):
        self.model     = model
        self.snapshots = deque(maxlen=max_snapshots)

    def save_snapshot(self):
        self.snapshots.append({
            k: v.clone().detach()
            for k, v in self.model.state_dict().items()
        })

    def predict_with_uncertainty(self, x):
        """Run forward pass for each snapshot → mean + std."""
        preds = []
        for snap in self.snapshots:
            self.model.load_state_dict(snap)
            with torch.no_grad():
                preds.append(self.model(x))
        preds = torch.stack(preds)   # [K, B, C]
        return preds.mean(0), preds.std(0)  # [B, C], [B, C]

# Training loop with cyclical temperature (Zhang et al. 2020)
def cyclical_sgld_train(model, train_loader, n_epochs=100,
                        cycle_len=10, T_max=1.0, T_min=0.0):
    ensemble = BayesianEnsemble(model)
    criterion= nn.CrossEntropyLoss()

    for epoch in range(n_epochs):
        # Cosine cycle: T starts high (exploration), ends near 0 (sampling)
        cycle_pos = epoch % cycle_len / cycle_len
        T         = T_min + 0.5*(T_max-T_min)*(1 + np.cos(np.pi*cycle_pos))
        optimizer = SGLDOptimizer(model.parameters(), lr=1e-3, temperature=T)

        for x, y in train_loader:
            optimizer.zero_grad()
            criterion(model(x), y).backward()
            optimizer.step()

        # At end of each cycle (low T), save parameter snapshot
        if (epoch+1) % cycle_len == 0:
            ensemble.save_snapshot()

    return ensemble

Mean-Field Theory & Fokker-Planck Equations

When you have millions of neurons, the law of large numbers turns stochastic dynamics into deterministic PDEs — explaining signal propagation, expressivity, and why deep networks can train at all.

From Individual Neurons to Population Dynamics

In a wide neural network (width N → ∞), the behavior of individual neurons becomes predictable. Each neuron's pre-activation is a sum of many independent contributions — by the Central Limit Theorem, it converges to a Gaussian. The entire network's forward pass can then be described by a deterministic PDE over the distribution of neuron activations rather than individual values.

The Fokker-Planck equation describes how a probability density evolves under an SDE. If the SDE governing neuron dynamics is dX = f(X,t)dt + σ(X,t)dW, then the density p(x,t) satisfies:

∂p/∂t = -∂(f·p)/∂x + (1/2)∂²(σ²·p)/∂x² (Fokker-Planck PDE)

This is why mean-field theory connects directly to the expressivity of neural networks: the order-chaos phase transition (Poole et al., 2016) — whether signals propagate or vanish through depth — is determined by properties of this PDE's solution. Networks at the "edge of chaos" (critical initialization) have the deepest trainable depth.

Fokker-Planck + Mean-Field Neural Network Theory
Fokker-Planck: ∂p/∂t = -∇·(f·p) + (1/2)∇²(σ²·p) Signal propagation in deep networks: q^l = E_{z~N(0,q^{l-1})} [φ(z)²] (mean-field variance recursion) where φ = activation, q⁰ = input variance "Edge of chaos": χ = σ_w² · E[φ'(z)²] = 1 (critical condition) χ < 1: ordered phase (signal decays, vanishing gradients) χ > 1: chaotic phase (signal explodes, diverging gradients) χ = 1: critical / "edge of chaos" (optimal depth for training)
PYTHON mean_field.py — signal propagation + edge of chaos analysis 
import numpy as np
import torch, torch.nn as nn

# ── Mean-field variance propagation ──────────────────────
def mean_field_variance(q0, sigma_w, sigma_b, n_layers,
                        activation=np.tanh):
    """
    Compute mean-field variance recursion across layers.
    q^l = σ_w² · E[φ(z)²] + σ_b²,   z ~ N(0, q^{l-1})
    Fixed point: q* satisfies q = σ_w²·E[φ(z)²] + σ_b²
    If q* is stable and χ(q*) = 1 → edge of chaos.
    """
    q = q0
    qs = [q]
    n_mc = 50_000  # MC samples for expectation
    for l in range(n_layers):
        z    = np.random.normal(0, np.sqrt(q), n_mc)
        phi_z= activation(z)
        q    = sigma_w**2 * np.mean(phi_z**2) + sigma_b**2
        qs.append(q)
    return np.array(qs)

def lyapunov_chi(q_star, sigma_w, activation_prime):
    """
    χ = σ_w² · E[φ'(z)²]  at the fixed point q*.
    χ < 1: ordered (vanishing gradients, good for shallow nets)
    χ = 1: edge of chaos (deepest trainable networks — BERT, GPT init)
    χ > 1: chaotic (exploding gradients)
    """
    z     = np.random.normal(0, np.sqrt(q_star), 100_000)
    dphi  = activation_prime(z)
    return sigma_w**2 * np.mean(dphi**2)

# Tanh derivative: φ'(z) = 1 - tanh²(z)
tanh_prime = lambda z: 1 - np.tanh(z)**2

# Find critical sigma_w for edge of chaos with tanh activation
for sigma_w in np.arange(0.5, 3.0, 0.1):
    qs  = mean_field_variance(1.0, sigma_w, 0.0, 200)
    q_star = qs[-1]
    chi = lyapunov_chi(q_star, sigma_w, tanh_prime)
    if abs(chi - 1.0) < 0.05:
        print(f"Edge of chaos: σ_w ≈ {sigma_w:.1f}, χ = {chi:.3f}")

# ── Fokker-Planck PDE via finite differences ─────────────
def fokker_planck_1d(f, D, x_grid, p0, dt=0.001, n_steps=1000):
    """
    Solve: ∂p/∂t = -∂(f·p)/∂x + D·∂²p/∂x²
    f: drift,  D: diffusion coefficient
    Models distribution of neural activations over time/depth.
    """
    dx = x_grid[1] - x_grid[0]
    p  = p0.copy()
    snapshots = [p.copy()]
    for _ in range(n_steps):
        fp    = f(x_grid) * p
        # Drift term: -∂(f·p)/∂x  (upwind scheme for stability)
        drift = -(np.roll(fp, -1) - np.roll(fp, 1)) / (2*dx)
        # Diffusion term: D·∂²p/∂x²  (central differences)
        diff  = D * (np.roll(p,-1) - 2*p + np.roll(p,1)) / dx**2
        p = p + dt * (drift + diff)
        p = np.maximum(p, 0); p /= p.sum() * dx  # normalize
        if _ % 100 == 0: snapshots.append(p.copy())
    return snapshots

Mean-Field Theory & Fokker-Planck — LLM Design Implications

  • Weight initialization (He / Xavier / LeCun): These initialization schemes are derived directly from mean-field signal propagation theory — they place the network at or near the edge of chaos (χ ≈ 1), enabling training of deep networks. GPT-2 uses σ_w = 0.02, GPT-3 scales inversely with depth — both derived from this theory.
  • LayerNorm placement: Pre-LN (normalize before attention) vs Post-LN (normalize after) is a direct trade-off in the Fokker-Planck evolution of activation distributions. Pre-LN keeps q_l ≈ 1 at every layer (ordered phase, more stable training), used in GPT-3 and LLaMA.
  • Attention entropy and rank collapse: Mean-field analysis of softmax attention shows that for deep transformers, attention weights collapse to uniform or spiked extremes — a phase transition in the PDE sense. Solutions: temperature scaling, ALiBi, or architectural changes like SwiGLU.
  • μP (maximal update parametrization): Greg Yang's μP (used to train Cerebras-GPT and Microsoft φ-1) uses mean-field theory to derive the correct learning rate scaling as width increases — enabling zero-shot hyperparameter transfer from small proxy models to trillion-parameter models.
  • Infinite-width limits (NTK and MF): At initialization, NTK theory (gradient flow linearization) and mean-field theory (activation distribution PDE) are two complementary infinite-width limits. Together they explain why transformers with skip connections and layernorm can be trained without careful per-layer tuning.
